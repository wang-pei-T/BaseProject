# 05.技术规格方案_V1.0

版本：V1.0  
日期：2026-04-24  
适用范围：与[04.功能规格_V1.0.md](file:///f:/workspace/BaseProject/doc/04.%E5%8A%9F%E8%83%BD%E8%A7%84%E6%A0%BC_V1.0.md)配套的技术实现方案（不包含性能压测、并发调优等内容）

---

## 1. 目标与范围

### 1.1 建设目标

- 基于前后端分离架构实现管理系统能力，后端提供统一 REST API，前端提供管理端界面与配置化能力承载。
- 覆盖用户/组织/角色/权限/菜单/操作点、字典、配置中心、文件与附件、消息中心（含发送侧公告）、审计/日志、平台侧租户管理与跨租户协助、平台账号与权限体系等能力。
- 技术侧保证可扩展性：后续支持数据库国产化适配（不局限 MySQL），对象存储可切换（MinIO/本地/云存储）。
- 为高可用部署预留缓存模式扩展：内存/Redis 两套缓存方式并存，仅对“需要共享”的数据启用 Redis。

### 1.2 范围边界

- 本文档描述实现架构、模块边界、关键技术选型、接口与数据规范、适配扩展点。
- 不包含：性能基准、压测指标、并发参数、容量规划等内容（如需可另出“性能与容量规格”）。

## 2. 技术栈与版本基线

### 2.1 后端

- JDK：8
- Spring Boot：2.7.*
- ORM：MyBatis Plus
- 数据库：MySQL 5.7（V1 默认）；预留国产化数据库适配扩展

- 对象存储：MinIO（S3 接口，V1 默认）；预留本地/云存储适配扩展

### 2.2 前端

- React：16
- Ant Design：作为基础 UI 组件库
- 配置化诉求：表单、语句、参数、流程、凭证、消息等的配置化建模与渲染（以模块化方式落地）

### 2.3 运行环境与配置

- 多环境：dev / test / uat / prod
- 配置载体：application-{env}.yml（可对接外部配置中心，V1 先以文件为基准）

## 3. 总体架构

### 3.1 前后端分离

- 前端：负责路由、页面渲染、交互、表单与配置化渲染、调用后端 API。
- 后端：负责鉴权、权限校验、业务逻辑、数据持久化、对象存储对接、审计与日志。

### 3.2 服务端分层建议

- controller：HTTP 协议适配层（入参校验、DTO 转换、返回统一响应）
- service：业务编排层（事务边界在此层定义）
- manager/domain：领域逻辑层（可选，用于复杂业务聚合与规则封装）
- repository/mapper：数据访问层（MyBatis Plus Mapper + XML/注解 SQL）
- integration：外部依赖适配（Redis、MinIO/存储、第三方接口）
- security：认证鉴权与权限拦截
- common：统一响应、异常、工具、枚举、常量

## 4. 前端技术方案（React16 + Ant Design）

### 4.1 工程目录建议

- src/api：接口请求封装（按模块拆分）
- src/pages：页面级组件（按模块拆分）
- src/components：通用组件（表格、表单构建器、弹窗、权限组件等）
- src/utils：工具（请求封装、权限判断、格式化、下载、路由守卫）
- src/config：环境配置、常量、权限点映射、菜单配置
- src/store（可选）：全局状态（登录态、用户信息、权限、主题）

### 4.2 API 调用与统一错误处理

- 统一封装 request：
  - 自动附加鉴权头（JWT 或 Cookie/Session）
  - 统一处理后端错误码：弹窗/提示/跳转登录/权限不足提示
  - 统一 requestId 展示与问题定位入口（可选）

### 4.3 权限驱动 UI

- 菜单与按钮权限：
  - 菜单：基于后端“我的菜单”接口返回树进行渲染，避免前端硬编码可见性逻辑。
  - 按钮/操作点：基于权限点集合（permissionCodes）进行渲染控制。
- 高危操作二次确认：
  - 删除/恢复、替换权限、强制下线等操作，前端统一确认弹窗与风险文案展示；平台侧跨租户动作要求 reason 必填并进入审计。

### 4.4 配置化能力落地建议

- 表单配置化：以 JSON Schema / 自定义 schema 描述字段、校验、联动、展示逻辑，渲染为 AntD Form。
- 语句/参数/流程/凭证/消息配置化：建议采用“配置定义（schema）+ 运行时解释器（renderer/executor）”模式：
  - schema 层负责结构与规则表达
  - renderer/executor 层负责渲染与执行编排

## 5. 后端技术方案（Spring Boot 2.7 + MyBatis Plus）

### 5.1 API 规范与统一响应

- API 风格：遵循[04.功能规格_V1.0.md](file:///f:/workspace/BaseProject/doc/04.%E5%8A%9F%E8%83%BD%E8%A7%84%E6%A0%BC_V1.0.md)中的 REST + 动作后缀约束。
- 统一响应格式：
  - 成功：{ code: "OK", message: "success", requestId, data }
  - 失败：{ code, message, requestId, details? }
- requestId：
  - 服务端生成并贯穿日志/审计；响应中返回给前端用于定位问题。

### 5.2 参数校验与异常处理

- 参数校验：统一使用 Bean Validation（javax.validation）对 DTO 进行校验，失败映射为 VALIDATION_ERROR。
- 统一异常处理：
  - 业务异常（BusinessException）：携带错误码与 message
  - 鉴权异常：AUTH_UNAUTHORIZED / FORBIDDEN
  - 资源不存在：*_NOT_FOUND
  - 唯一冲突：CONFLICT_UNIQUE（由数据库异常转换统一识别）

### 5.3 认证与会话

- 支持两种登录模式：
  - JWT：token 通过 Authorization: Bearer 传递
  - Session：cookie/sessionId 模式
- 会话管理：
  - 支持会话列表查询与下线指定会话（对应功能规格 SESS-001/SESS-002）
  - 多实例部署时，会话数据需可共享（建议使用 Redis 存储会话态或撤销表）

### 5.4 权限体系（RBAC）

- 权限点命名与字典表：以功能规格 4.0 “权限点字典表”为唯一准绳。
- 鉴权实现建议：
  - Controller 层使用注解或声明式配置：@RequirePermission("tenant.user.create")
  - Security 统一拦截并从上下文获取当前主体权限集合
- 平台侧账号与权限体系：
  - 平台账号、平台角色、平台权限点与租户侧体系隔离，分别校验 scope=platform/tenant 的权限点。

### 5.5 数据权限

- 数据隔离维度：
  - tenantId：租户强隔离（租户侧所有业务表必须包含 tenant_id）
  - orgId（部门/组织）：用于数据范围过滤（若启用）
- 落地方式：
  - service 层统一注入过滤条件（推荐，便于审计与排障）
  - 或使用 MyBatis Plus 拦截器统一注入（需要谨慎限定生效范围）

## 6. 数据库设计与适配扩展（MySQL 5.7 默认，预留国产化适配）

### 6.1 数据库选型原则

- V1：MySQL 5.7
- V2+：支持切换/兼容国产数据库（如达梦、人大金仓、OceanBase、TiDB、GaussDB 等），以“公共 SQL 规范 + 方言隔离 + 兼容测试”控制改造成本。

### 6.2 公共 SQL 规范（为跨库做约束）

- 禁止依赖强 MySQL 方言特性：
  - 不使用 INSERT ... ON DUPLICATE KEY UPDATE / REPLACE INTO 作为核心流程唯一实现
  - 不在业务核心依赖 MySQL JSON 函数与 JSON 字段特性
  - 不手写 LIMIT 方言分页（分页交给 MyBatis Plus）
- 函数与表达式：
  - 优先使用标准 SQL：COALESCE、CASE WHEN 等，避免 IFNULL、FIND_IN_SET 等方言函数
- 标识符与保留字：
  - 表/字段命名避免保留字；需要引用时通过方言包装处理（见 6.4）

### 6.3 MyBatis Plus 使用约束

- 业务查询优先使用 LambdaQueryWrapper/LambdaUpdateWrapper，减少手写 SQL。
- 复杂 SQL 如需 XML：
  - 限制在少量 mapper.xml 中
  - 所有方言分支集中使用 <choose> 管理，不允许分散在业务代码中

### 6.4 方言隔离层（DbDialect）

为跨数据库适配定义方言抽象，集中处理差异点：

- DbDialect 能力建议：
  - getDbType：mysql/dm/kingbase/oracle/postgresql 等
  - wrapIdentifier：标识符引用方式（`、" 等）
  - isDuplicateKeyException：统一识别唯一冲突异常并映射 CONFLICT_UNIQUE
  - buildLike：LIKE 模糊查询差异（如大小写/转义规则差异，必要时处理）
- 数据库类型识别：
  - 运行时读取 DatabaseMetaData#getDatabaseProductName 并缓存
  - 配置项提供兜底：app.db.dialect=mysql

### 6.5 ID、时间与字段类型规范

- 主键：
  - 建议 Java 侧生成（雪花/UUID），避免依赖 AUTO_INCREMENT 以降低跨库差异
- 软删除：
  - 统一使用 deleted_at/deleted_by（或 MyBatis Plus 逻辑删除字段）
- 乐观锁：
  - 统一 version 字段（配置项等需要并发写保护的场景）
- 字段类型建议（尽量跨库）：
  - string：varchar
  - text：text/clob（按方言映射）
  - 时间：datetime/timestamp（统一序列化规范）

## 7. 缓存与高可用适配（内存/Redis 双模式）

### 7.1 设计目标

- 单机/开发环境可仅使用内存缓存。
- 高可用部署时，对“需要共享”的数据使用 Redis；对“无需共享”的热点数据可继续使用本地缓存。

### 7.2 CacheProvider 抽象

- CacheProvider：get/set/del/mget/ttl 等基础能力
- 实现：
  - InMemoryCacheProvider：本地内存（如 ConcurrentHashMap 或 Caffeine）
  - RedisCacheProvider：Redis
- 模式配置：
  - app.cache.mode=memory|redis|hybrid

### 7.3 需要共享的数据范围（建议）

- 会话/登录态（Session 或 JWT 撤销表/黑名单）
- 幂等键（Idempotency-Key）
- 权限聚合缓存（用户权限集合、角色权限版本号）

## 8. 对象存储适配（MinIO / 本地 / 云存储）

### 8.1 目标

- V1 默认对接 MinIO（S3 API）。
- 后续可替换为：
  - 本地文件系统（单机/共享盘模式）
  - 云对象存储（OSS/COS/OBS/S3 等）
- 业务层不依赖具体对象存储实现细节（URL 格式、SDK 差异）。

### 8.2 StorageProvider 抽象

建议定义统一对象存储接口，提供最小闭环能力：

- putObject(stream, objectKey, contentType, size, metadata)
- getObject(objectKey)
- deleteObject(objectKey)
- exists(objectKey)
- createPresignedUrl(objectKey, expirySeconds, method)（可选；本地存储可返回后端受控下载地址）

Provider 实现：

- S3StorageProvider：MinIO/S3 兼容云存储
- LocalStorageProvider：本地目录存储
- CloudStorageProvider：按云厂商扩展（OSS/COS/OBS）

### 8.3 文件元数据与附件关系建模

- file（文件元数据表）：
  - file_id、storage_type、bucket、object_key、size、content_type、etag/hash、created_by、tenant_id、created_at
- attachment（附件表，业务引用）：
  - attachment_id、biz_type、biz_id、file_id、created_by、tenant_id、created_at、deleted_at

关键原则：

- 外部可访问 URL 不入库（仅在响应中短期返回）。
- 对外下载/预览通过：
  - S3/MinIO：短期 presigned URL
  - 本地：后端受控下载接口（鉴权后流式输出）

### 8.4 本地存储在 HA 下的约束

- storage.type=local 仅适用于：
  - 单机部署，或
  - 多实例共享盘（NFS/分布式文件系统）模式
- 技术方案中以配置强约束：
  - app.storage.local.mode=single|shared
  - shared 模式必须配置共享挂载目录 app.storage.local.baseDir

## 9. 文件上传下载与安全策略（与功能规格对齐）

- 上传入口：FILE-001
  - 校验：类型、大小、登录态、tenantId
  - 入库：先写 file 元数据，再写对象存储
- 下载入口：FILE-002
  - 校验：资源可见性（tenantId/业务对象权限）
  - 返回：直出流或 presigned URL
- 预览入口：FILE-003
  - 返回：短期 URL 或受控预览流
- 附件绑定：ATT-001/ATT-002/ATT-003/ATT-004
  - bizType/bizId 的可见性校验由业务模块提供（或统一回调接口）

## 10. 消息中心与平台公告（收件侧 + 发送侧）

### 10.1 收件侧

- MSG-001/MSG-002/MSG-003/MSG-004：列表、详情、已读、全部已读

### 10.2 发送侧（平台公告）

- PANN-001~PANN-005：草稿、发布、撤回、管理端列表/详情
- 发布策略：
  - 发布后为目标租户/用户生成消息投递记录（作为收件侧数据来源之一）
- 审计：
  - 创建/发布/撤回必须进入审计，撤回 reason 必填（如功能规格约束）

## 11. 审计与日志

### 11.1 审计

- 审计表记录字段建议：
  - eventCode、operatorId、operatorType（tenantUser/platformAccount）、tenantId（若有）、targetType/targetId、requestId、result、createdAt、details（hash/摘要）
- 敏感字段：
  - 密码/token/临时 URL 等不得记录明文
  - phone/email/idCard 等按功能规格“数据脱敏规则”执行

### 11.2 业务日志与访问日志

- 采用 logback-spring.xml 配置
- 关键字段：
  - requestId、tenantId、userId/accountId、uri、method、status、elapsed

## 12. 配置管理与多环境

### 12.1 配置分层

- application-{env}.yml：基础运行参数
- 配置中心（业务配置项）：
  - 平台配置：/platform/configs
  - 租户配置：/tenant/configs
  - 租户覆盖项：TCONF-001~TCONF-003（仅允许覆盖白名单 key）

### 12.2 关键配置项建议（示例）

- app.auth.mode=jwt|session
- app.cache.mode=memory|redis|hybrid
- app.db.dialect=mysql|dm|kingbase|...
- app.storage.type=s3|local|oss|cos|obs
- app.storage.s3.endpoint / accessKey / secretKey / bucket
- app.storage.local.baseDir / mode
- app.security.riskConfirm.enabled=true
