## 1. 文档范围与版本

### 1.1 文档范围

本文档用于将 PRD 中的需求拆解为“可交付、可开发、可测试”的功能规格条目。每条功能规格遵循以下原则：

- 一条功能规格只描述一个可交付功能（不混多个动作）
- 每条规格可被开发与测试直接执行（避免“支持/优化/完善”等不可验收表述）
- 每条规格必须绑定：权限点 + 审计点 + 验收标准

### 1.2 版本信息

- 文档版本：V1.0
- 适用范围：平台侧（Platform Console）与租户侧（Tenant Console）中与本版本功能清单对应的条目

### 1.3 优先级与发布范围

优先级定义：

- P0：核心链路必做；无替代方案；阻塞上线
- P1：重要能力必做；允许短期绕行；不阻塞核心链路
- P2：增强能力可选；不影响核心可用性
- P3：后续版本能力；V1 不承诺交付

发布范围冻结规则（用于本版本内口径统一）：

- V1必做：P0、P1
- V1可选：P2
- V2：P3

## 3. 功能规格明细（逐条）

### AUTH-001

功能编号：AUTH-001
功能名称：JWT 登录
所属模块：认证与会话
业务目标：用户通过账号密码登录，获取可用于后续请求的 JWT 访问令牌（及可选刷新令牌），并建立当前登录态。

角色与权限点：

- 角色：匿名用户
- 权限点：无（公开接口）

前置条件：

- 租户可用（未停用、未到期）或符合配置允许登录的策略
- 用户账号存在，且未禁用、未锁定、未强制阻断（如需强制改密则按规则处理）

输入字段与校验规则（必填、长度、唯一性）：

- tenantCode（可选）：string，长度 1-64；若系统采用“租户域名/自动识别租户”，则可不传
- username：string，必填，长度 3-64
- password：string，必填，长度 8-128
- captchaToken（可选）：string；当配置开启验证码时必填

主流程（步骤化）：

1) 客户端调用 JWT 登录接口并提交 username/password
2) 服务端解析租户上下文（tenantId）
3) 校验租户状态，校验账号状态
4) 校验密码（不区分“账号不存在/密码错误”的对外提示）
5) 登录成功后生成 accessToken（必选）与 refreshToken（可选）
6) 返回 token 与用户基础信息

异常流程（失败分支）：

- 租户上下文缺失：返回 TENANT_CONTEXT_MISSING
- 租户停用/到期：返回 TENANT_DISABLED / TENANT_EXPIRED
- 用户被禁用：返回 USER_DISABLED
- 登录失败次数超限导致锁定：返回 USER_LOCKED
- 验证码缺失/无效：返回 CAPTCHA_REQUIRED / CAPTCHA_INVALID
- 账号或密码错误：返回 AUTH_INVALID_CREDENTIALS

接口定义（请求/响应/错误码）：

- POST /auth/login
- 请求头：
  - X-Auth-Mode: jwt
  - Idempotency-Key（可选）
- 请求体：
  - tenantCode?: string
  - username: string
  - password: string
  - captchaToken?: string
- 响应体：
  - accessToken: string
  - tokenType: "Bearer"
  - expiresIn: number（秒）
  - refreshToken?: string
  - user: { userId: string, displayName: string, username: string }
- 错误码：见 4.1

状态变化（前置状态 -> 后置状态）：

- 用户状态不变；更新用户最近登录时间、登录失败计数（成功归零，失败递增）

审计要求（记录哪些字段）：

- 安全审计事件：LOGIN_SUCCESS / LOGIN_FAILURE
- 记录字段：operatorType（anonymous）、tenantId、username（或 userId 若已定位）、result、ip、userAgent、requestId

安全要求（越权、跨租户、敏感操作）：

- 不泄露账号是否存在（统一提示）
- 强制 tenantId 隔离：不得在无 tenantId 上下文时继续验证
- 密码不得出现在日志与审计中

验收标准（可测试）：

- 正确账号密码返回 accessToken，且 /auth/me 可获取当前用户信息（若存在该接口）
- 错误密码返回 AUTH_INVALID_CREDENTIALS，且不泄露账号是否存在
- 达到失败阈值后账号锁定并返回 USER_LOCKED
- 审计中可查询到登录成功/失败记录，且包含 requestId

备注（可选）：

- refreshToken 若启用需支持撤销与轮换策略（不在本条规格展开）

### AUTH-002

功能编号：AUTH-002
功能名称：Session 登录
所属模块：认证与会话
业务目标：用户通过账号密码登录，服务端创建会话并通过 Cookie（或等价机制）维持登录态。

角色与权限点：

- 角色：匿名用户
- 权限点：无（公开接口）

前置条件：

- 同 AUTH-001

输入字段与校验规则（必填、长度、唯一性）：

- 同 AUTH-001

主流程（步骤化）：

1) 客户端调用 Session 登录接口提交 username/password
2) 服务端解析 tenantId 并校验租户与账号状态
3) 校验密码
4) 创建服务端 Session（存储到配置的会话存储）
5) 通过 Set-Cookie（或等价机制）返回 sessionId
6) 返回用户基础信息

异常流程（失败分支）：

- 同 AUTH-001

接口定义（请求/响应/错误码）：

- POST /auth/login
- 请求头：
  - X-Auth-Mode: session
- 请求体：同 AUTH-001
- 响应：
  - Set-Cookie: sessionId=...; HttpOnly; Secure; SameSite=...
  - body: { user: { userId, displayName, username } }
- 错误码：见 4.1

状态变化（前置状态 -> 后置状态）：

- 用户状态不变；更新最近登录与失败计数

审计要求（记录哪些字段）：

- 同 AUTH-001（LOGIN_SUCCESS / LOGIN_FAILURE）

安全要求（越权、跨租户、敏感操作）：

- Cookie 必须 HttpOnly；Secure 依据部署环境启用；SameSite 策略按配置
- 会话固定攻击防护：登录成功后必须生成新的 sessionId

验收标准（可测试）：

- 登录成功响应携带 Set-Cookie，后续请求携带 Cookie 可通过鉴权
- 登录失败不返回有效 sessionId
- 审计记录包含登录结果与 requestId

备注（可选）：

- CSRF 防护策略属于全局安全策略（不在本条规格展开）

### AUTH-003

功能编号：AUTH-003
功能名称：注销
所属模块：认证与会话
业务目标：终止当前登录态（JWT 模式撤销当前令牌或加入黑名单；Session 模式销毁当前会话）。

角色与权限点：

- 角色：已登录用户
- 权限点：无（对已登录用户开放）

前置条件：

- 当前请求已通过鉴权

输入字段与校验规则（必填、长度、唯一性）：

- 无

主流程（步骤化）：

1) 客户端调用注销接口
2) 服务端识别当前认证模式（JWT/Session）
3) JWT：将当前 token 标记为失效（策略：黑名单/撤销表/短期缓存，按实现）
4) Session：销毁当前 session
5) 返回成功

异常流程（失败分支）：

- 未登录调用：返回 AUTH_UNAUTHORIZED

接口定义（请求/响应/错误码）：

- POST /auth/logout
- 响应体：{ success: true }
- 错误码：见 4.1

状态变化（前置状态 -> 后置状态）：

- 登录态终止；用户状态不变

审计要求（记录哪些字段）：

- 安全审计事件：LOGOUT
- 记录字段：operatorType、tenantId、userId、ip、userAgent、requestId

安全要求（越权、跨租户、敏感操作）：

- 只允许注销当前登录主体

验收标准（可测试）：

接口命名规范（V1 冻结）：

- 租户侧接口统一以 /tenant 为前缀；平台侧接口统一以 /platform 为前缀

### 4.2 分页协议

统一采用 Page 分页：

- 请求参数：page（>=1）、pageSize（1-200）、sortBy、sortOrder
- 响应字段：items、page、pageSize、total

### 4.3 时间规范

- 所有时间字段使用 ISO 8601（RFC 3339）格式，必须携带时区，例如：2026-04-24T10:20:30+08:00
- 服务端审计与日志记录必须包含 requestId，可与链路追踪关联（若启用）

### 4.4 幂等规范

- 对于状态切换类动作（enable/disable/lock/unlock/restore 等），若目标已处于期望状态，应返回成功（幂等）
- 对于创建类接口（如 USER-001），可选支持 Idempotency-Key：
  - 相同 Idempotency-Key 在有效期内重复提交应返回相同结果或可识别的冲突结果

### 4.4.1 权限冲突与计算顺序（V1 冻结）

权限来源定义：

- allow：来自“角色权限集合”（ROLE-006）与“菜单访问权限绑定”（MENU-006）等授权配置产生的允许项
- deny：来自“受保护的系统拒绝规则”（系统策略）与“主体级拒绝清单”（黑名单/冻结）产生的拒绝项

deny 来源（V1 冻结）：

- 系统策略 deny：
  - 用户被锁定（USER_LOCKED）
  - 用户被禁用（USER_DISABLED）
  - 用户被软删除（USER_DELETED）
  - 租户停用/到期（TENANT_DISABLED/TENANT_EXPIRED）
  - 角色被禁用（ROLE_DISABLED，禁用角色产生的 allow/deny 均不参与计算）
- 主体级 deny（可选，V2）：用户/账号黑名单（不在 V1 范围）

计算顺序（V1 冻结）：

1) 校验租户状态（停用/到期直接拒绝）
2) 校验用户状态（禁用/锁定/删除直接拒绝）
3) 计算用户有效角色集合（忽略 disabled/deleted 角色）
4) 汇总 allow 集合（角色权限集合 union）
5) 如存在 deny 集合（仅系统策略在前置步骤体现），deny 优先于 allow

输出约束：

- PERM-002 / PASSIST-002 追溯结果必须体现“拒绝原因”的优先级（例如 tenantDisabled/userLocked/roleDisabled）
- “我的菜单”（MENU-007）必须与权限计算一致：权限拒绝导致菜单不可见

### 4.5 批量接口失败策略（V1 冻结）

V1 默认采用全成全败（原子性）策略：

- 批量接口（如移动用户到机构、排序调整）若任一条记录失败，整体失败；不落库任何部分结果
- 响应错误码按失败类型唯一返回（VALIDATION_ERROR / NOT_FOUND / FORBIDDEN / CONFLICT_UNIQUE 等）
- details 提供失败原因与定位信息（用于前端定位与测试核对）：
  - { index, field?, code, message, entityId? }

V2 可扩展为部分成功策略（不在 V1 范围内）：

- 若启用部分成功，必须在接口响应体中返回 succeededIds / failedItems，且失败项包含 code 与 message

### 4.6 数据脱敏规则（V1 冻结）

脱敏适用范围：

- 返回给前端：列表/详情
- 持久化记录：应用日志/访问日志/审计日志

脱敏总则：

- 列表优先脱敏；详情按权限与最小暴露原则脱敏
- 日志/审计默认不记录明文敏感值；确需记录时仅记录 hash/后四位/掩码

字段脱敏规则表（V1.0）：

| 字段 | 列表返回 | 详情返回 | 应用/访问日志 | 审计日志 |
| --- | --- | --- | --- | --- |
| password / initialPassword | 不返回 | 不返回 | 不记录 | 不记录 |
| refreshToken / accessToken / sessionId | 不返回 | 不返回 | 不记录 | 不记录 |
| phone | 掩码（3-4-4） | 掩码（3-4-4） | 不记录明文（可记录后四位） | 不记录明文（可记录后四位） |
| email | 掩码（@ 前保留 1-2 位） | 掩码（@ 前保留 1-2 位） | 不记录明文 | 不记录明文 |
| idCard / bankCard（若未来引入） | 掩码（仅后四位） | 掩码（仅后四位） | 不记录明文 | 不记录明文 |
| fileUrl / downloadUrl（临时 URL） | 不返回 | 返回（短期有效） | 不记录完整 URL（可记录域名与路径 hash） | 不记录完整 URL（可记录 hash） |
| reason / 工单号（跨租户协助） | 返回 | 返回 | 记录 | 记录 |

### 4.7 并发、事务与原子性边界（V1 冻结）

- 角色权限替换（ROLE-006）：必须原子替换；任一权限点/操作点不存在则整体失败并回滚
- 批量移动用户到机构（ORG-008）：必须原子移动；任一用户不存在/越权/目标机构不可用则整体失败并回滚
- 排序调整（MENU-003、DICTI-003）：必须原子更新排序；传入列表必须同一资源集合，否则 VALIDATION_ERROR
- 配置更新（CONF-002）：更新必须基于 expectedVersion 乐观锁；版本冲突返回 CONFIG_VERSION_CONFLICT（409），不写入
